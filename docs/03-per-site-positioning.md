# Per-Site Positioning

*Companion to `01-brand-foundation.md` and `02-visual-identity.md`. For each of the four properties: role, audience, tagline, hero copy, value prop, primary CTA, tone notes, and quick-fix observations from the live site.*

---

## kochie.io — the front door

**Role in the system.** The umbrella. First-time visitors who don't know Rob personally land here. Its job is to (a) say what Kochie Engineering is, (b) prove it with shipped products, (c) route the visitor to the sub-property that matches their intent.

**Primary audience.** Engineers who landed via a blog post, a product, or a referral and want to know "wait, who is this person/studio?".

**Secondary audience.** Conference organisers, hiring managers, podcast hosts doing a 60-second sniff test.

### Tagline

> **A studio of one. Melbourne.**

Direct, specific, slightly understated. Doesn't oversell. Locates you.

*Alternates if you want to A/B against:*
- *Engineering, with the working out shown.*
- *Small software, properly built.*

### Hero copy

> **A studio of one. Melbourne.**
>
> Rob Koch builds small, sharp software and writes about how it gets made.
>
> [See what's shipped →]   [Read the blog →]

That's the whole hero. Two CTAs because the two highest-value paths are (1) products, (2) blog. A "Get in touch" CTA belongs further down, not in the hero — it pre-filters out the people you actually want.

### Value prop

What Kochie Engineering offers visitors:

1. **Shipped products** they can try (Touch Typer, MetaFixer).
2. **Substantive writing** they can learn from (the blog).
3. **A real engineer** they can talk to if there's a fit.

### Primary CTA

`See what's shipped` → links to a Products section showcasing Touch Typer + MetaFixer + future products. Not a contact form. Contact is a tertiary action.

### Tone notes

- Confident, factual, not promotional.
- No "we" — there is no we. Either say "Kochie Engineering" (when speaking as the studio) or "Rob" (when the human is in the frame). Pick one per page and stay there.
- Mono kicker above hero: `// kochie.io`.

### Quick fixes for the current site

The live kochie.io has a positioning problem that's worth being direct about:

| Currently says | Why it's off | Replace with |
|---|---|---|
| "Web Development, Made Simple." | Generic web-agency framing; says nothing about who you are | "A studio of one. Melbourne." |
| "We've helped businesses of all sizes…" | "We" implies a team that doesn't exist; "businesses of all sizes" is filler | "A few things I've shipped recently." |
| "Our streamlined process makes web development simple, transparent, and stress-free" | Reads like every freelancer's homepage on Upwork | Cut entirely or replace with "How I work: small scope, fast feedback, real numbers." |
| "Global Support / 24/7 / Multilingual team" with country flags | This is a fiction. It will undermine credibility with the actual peer audience the moment they read it | Cut the entire section. Replace with a "Now" block: what you're building, reading, talking about this month |
| "Our team offers support in multiple languages and timezones" | See above | Remove |
| Footer: "Services / UI/UX Design / Mobile Apps / Cloud Solutions / Consulting" | Five service lines for a one-person studio reads as overreach | Replace with: `Products`, `Writing`, `Talks`, `About`. That's the structure of the brand |

The single most valuable change you can make on kochie.io: **delete every plural pronoun and every claim of being a team.** It will feel scary; it will read more credible to your actual audience.

## me.kochie.io — the human

**Role in the system.** Rob the person. Bio, work history, projects, talks, way to get in touch. The one place in the system where first person is the *only* allowed voice.

**Primary audience.** People deciding whether to work with, talk to, hire, or invite Rob.

**Secondary audience.** Curious peers who clicked through from the blog or a talk.

### Tagline

> **Engineer. Melbourne. Currently shipping at Redactive.**

Three things people actually want to know. Update the third when it changes.

*Existing line is also strong — keep it as the secondary line under the H1:*  
*"I build things, write the code for them, and run from the resulting explosion."*

### Hero copy

> # I'm Rob.
>
> Engineer. Melbourne. Currently shipping at Redactive AI on AI security.
>
> *I build things, write the code for them, and run from the resulting explosion.*
>
> [What I've built →]  [Read the blog →]  [Book a call →]

### Value prop

This site doesn't sell a service — it builds trust. The "value" is *evidence*: bio, employers, projects, talks, awards. Lead with the recent / heaviest credentials (Redactive, AWS, Localz), let the rest unfold below.

### Primary CTA

`Book a call` is already there (cal.com/kochie) — keep it as the primary "I want to talk to Rob" path. Email as the fallback.

### Tone notes

- First person, always.
- Warm but not cute. The runner-bouldering-karate-gaming list is great — leaves it on the page; don't dial it down.
- The "scream into the void" newsletter line is on-brand voice — keep it but maybe give the newsletter itself a real name (see Foundation §10).

### Quick fixes for the current site

- **Reorder the bio paragraphs.** Lead with the current role (Redactive), not the chronological "I'm a graduate of Monash". The "Right now I'm a committee member" paragraph is buried — surface it sooner if Young Engineers Australia is part of how you want to be known.
- **Consolidate "Recent Work" with kochie.io's projects section.** The same projects appear on both. Pick one as canonical (recommend kochie.io) and have me.kochie.io link to it: "I've shipped a few things — see the studio →"
- **Talks section is great. Keep it.** Add a `Speaking?` line at the bottom: "Looking for a speaker on AI security or AWS internals? [Get in touch]."
- **Holopin badges section** can stay — it's quirky, on-brand for the dev-peer audience, and zero-cost to keep.

## blog.kochie.io — the thinking

**Role in the system.** The thinking-out-loud surface. Long-form, technical, opinionated. This is the site that earns you organic traffic, dev-peer credibility, and inbound contact.

**Primary audience.** Working engineers searching for a real answer to a specific problem (lambda recursion, S3 limits, Electron + StoreKit, etc.).

**Secondary audience.** Subscribed readers who want Rob's takes. Think "former colleague who's interested in what you're up to."

### Tagline

> **Field notes from a one-person engineering studio.**

Or, if you want the shorter, voice-ier option:

> **Software, mostly. Sometimes maths.**

(Both work. The first is more searchable / shareable; the second is more on-brand voice.)

### Hero copy

> # Kochie Engineering / Blog
> Field notes from a one-person engineering studio. Software, mostly. Sometimes maths.
>
> [Latest posts →]  [Subscribe →]

### Value prop

Two reasons to read:

1. **Specific technical answers** that are deeper than the average tutorial. (Real data, real benchmarks, real war stories.)
2. **A consistent voice** — the same engineer thinking about different things, over time. The kind of blog where you might read a post about Halo physics next to one about S3 internals and trust both.

### Primary CTA

`Subscribe` (newsletter or RSS — keep both prominent). Direct conversion goal: turn a one-time Googled-here visitor into a returning reader.

### Tone notes

- First person.
- Allowed to be opinionated, even slightly grumpy. "Stop writing recursive Lambdas" is the energy.
- Earn the take with the working out. Show the code, the diagram, the benchmark.
- Tags are part of the voice — the existing `100DTO`, `maths`, `cdk`, `gaming` set is good. Keep them visible.

### Quick fixes for the current site

- **Add a one-line "what is this" subhead under the H1.** Right now the homepage is just a logo + a list of articles; visitors who arrive via Google to a single post and click the logo land somewhere ambiguous. The tagline above fixes it.
- **Surface the author.** The footer has "Robert Koch" but the homepage doesn't. Add a one-line `By Rob Koch — engineer, Melbourne. [More about Rob →]` below the H1.
- **Mono article numbers.** The URLs already encode them (`13-lambda-recursion`); show the number as a `// 13` mono kicker on each card. Tiny change, big "this is a coherent system" payoff.
- **Add "Updated" dates** in mono next to "Published" dates for the evergreen technical posts. Increases trust and search ranking.

## touch-typer.kochie.io — the product

**Role in the system.** The first shipped product. Should stand on its own as a download landing page, while clearly belonging to the Kochie Engineering family.

**Primary audience.** People who want to learn touch typing or get faster at it. Mostly developers, students, and people switching keyboard layouts (Dvorak / Colemak).

**Secondary audience.** Open-source-curious devs who arrived via the GitHub repo or a blog post.

### Tagline

> **Practice typing. See it stick.**

The current "Learn at your own pace" is fine but generic. The above is more specific to the product's actual value (the analytics / tracking-improvement angle is differentiating).

*Alternates:*
- *A typing tutor that respects your time.*
- *Get faster. See the proof.*

### Hero copy

> # Practice typing. See it stick.
>
> Touch Typer is a free, open-source typing tutor for Linux, Windows, and macOS. Track your speed, accuracy, and improvement over time. Learn a new keyboard layout, or just stop hunting for the comma.
>
> [Download →]  [See the leaderboard]
>
> *Free. Open source. Made in Melbourne.*

### Value prop

What Touch Typer actually offers, in one sentence each:

1. **Cross-platform** — Linux, Windows, macOS, App Store + Snap Store.
2. **Free and open source** — no upsell, no analytics-as-a-product.
3. **Real progress tracking** — speed, accuracy, error patterns over time. This is the differentiator vs. typing.com / monkeytype.
4. **Layout-agnostic** — useful for anyone trying Dvorak, Colemak, or just trying to get above 60wpm.

### Primary CTA

`Download` — anchor straight to the download section. Secondary: `Leaderboard` (creates a return-visit hook for power users).

### Tone notes

- Second person, helpful, plain. Different register from the rest of the system because the audience is different (not all visitors here are devs).
- Don't be cute. Typing tutors that try to be funny age badly.
- *Do* keep the "Made with [heart] in Melbourne" sign-off in the footer. It humanises the product and quietly cues the umbrella brand.

### Quick fixes for the current site

- **Add Kochie Engineering attribution in the footer.** One line: `Touch Typer is a product of Kochie Engineering. [More from Rob →]` linking to kochie.io. This is the architectural change that makes the family system real.
- **The fake user dropdown ("Tom Cook / tom@example.com")** appears to be a placeholder showing through to production. Either wire it up to real auth or remove it from the public landing page — it currently undermines credibility.
- **The hero photo is a stock-y face crop.** Replace with an actual screenshot of Touch Typer in action. Product shots beat lifestyle shots for utility apps every time.
- **"Copyright 2022"** in the footer reads as abandoned. Make it dynamic to current year, or remove the year entirely.
- **Link out to the GitHub repo** prominently. The "open source" claim is a credibility multiplier — only if visitors can find the code in one click.

## Cross-site consistency checklist

After all four sites are updated, every page across the system should hit these:

- [ ] Master logo lockup in the header (or a sub-brand lockup using the slash convention)
- [ ] Same font stack (Geist Sans / Geist Mono / Inter for blog body)
- [ ] Same ink/paper colour foundation; only the accent differs
- [ ] Footer line linking back to the umbrella: `Part of Kochie Engineering →` (where applicable)
- [ ] No "we" pronouns unless there literally is a we
- [ ] No "world-class / leverage / empower / seamless / robust" — see Foundation §5 retired-words list
- [ ] Mono used for metadata, code, and small accents — not as decoration
- [ ] One primary CTA per page (visually dominant), one secondary

If a page fails any of those checks, it's not yet on-brand.
